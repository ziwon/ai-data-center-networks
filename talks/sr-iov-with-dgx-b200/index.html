core terms
  • PF는 물리 NIC의 parent function이다. 포트, 펌웨어, VF 생성 정책을 가진다.
  • VF는 Pod에 넘겨줄 수 있는 가벼운 PCIe function이다. VM이나 container에서 직접 장치처럼 보인다.
  • SR-IOV CNI는 VF를 만들지 않는다. 선택된 VF를 Pod network namespace로 옮기고 IP를 설정한다.
  • Device Plugin은 VF 목록을 Kubernetes resource로 광고하고 kubelet allocation에 참여한다.
  • Multus는 기본 CNI 옆에 SR-IOV 보조 인터페이스를 붙이는 호출자 역할이다.

SR-IOV flow
what to verify
SR-IOV 문제를 볼 때는 “network attachment”와 “device allocation”을 분리한다
network attachment path
  • NAD가 어떤 secondary network를 붙일지 정의한다.
  • Multus가 Pod annotation을 읽고 SR-IOV CNI를 호출한다.
  • SR-IOV CNI가 선택된 VF netdev를 Pod namespace로 옮기고 IPAM을 적용한다.
device allocation path
  • Device Plugin이 VF를 Kubernetes extended resource로 광고한다.
  • Scheduler는 해당 resource가 남아 있는 node에 Pod를 배치한다.
  • Kubelet Device Manager가 concrete PCI address를 할당한다.

DGX B200 / IB 관점: VF가 Pod에 들어간 뒤에도 RDMA device, GUID, LID, Subnet Manager, mlx5_ib binding이 맞아야 실제 RDMA path가 열린다.

operational guardrailsfailure-prone edges
SR-IOV 운영에서 반드시 분리해서 볼 4가지
  • 기본 Pod network와 RDMA network는 목적이 다르다. eth0는 service/control traffic, net1은 high-performance data path로 설명한다.
  • Multus는 자동으로 RDMA fabric을 완성하지 않는다. 보조 인터페이스를 붙이는 호출 흐름을 제공할 뿐이다.
  • Whereabouts는 IPAM이다. IP를 할당하지만 LID/GUID/SM 문제를 해결하지 않는다.
  • Device Plugin은 scheduling resource를 만든다. CNI가 어느 VF를 써야 하는지 알 수 있게 kubelet allocation과 연결된다.
  • Pod에 VF가 보이는 것과 RDMA 가능 상태는 다르다. ibv_devinfo, LID, GUID, driver binding을 확인한다.
  • GPU locality가 빠지면 성능 문제가 남는다. GPU와 VF가 같은 rail/local NIC에 가까운지 확인한다.

한 문장으로: SR-IOV on K8s는 “NAD로 network를 고르고, Device Plugin으로 VF resource를 예약하고, kubelet과 CNI가 선택된 VF를 Pod namespace로 옮기는 흐름”이다.

closing messages
RDMA-ready는 Kubernetes와 InfiniBand 양쪽에서 동시에 검증된 상태다
  • AI 데이터센터 네트워크는 서버 네트워크가 아니라 GPU fabric이다.
  • DGX B200은 topology가 있는 system이다. OSFP, NIC, RDMA device, GPU affinity를 같이 본다.
  • Kubernetes는 fabric을 자동으로 이해하지 못한다. Cilium, Multus, SR-IOV, Device Plugin, scheduler를 연결해야 한다.
  • VF 생성과 RDMA 가능 상태는 다르다. GUID, LID, SM, mlx5_ib까지 확인한다.
  • 운영의 본질은 관측 가능성이다. 어느 레이어에서 깨졌는지 빠르게 식별해야 한다.

Operational rule: VF는 Kubernetes에 보이고, Linux에서 bind되고, IB fabric에서 LID/GUID를 받고, GPU path와 locality가 맞을 때 RDMA-ready로 본다.
